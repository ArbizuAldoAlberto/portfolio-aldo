# El Problema de las Ventas Perdidas

En entornos hostiles (sótanos de hormigón, campos remotos, camiones en ruta), la conectividad celular es un lujo, no una garantía. Cuando un usuario de tu aplicación logística o de ventas no puede registrar una transacción por falta de señal, no solo pierde tiempo: **tú pierdes dinero**.

## La Falacia del "Online-First"
La mayoría de las aplicaciones móviles están diseñadas con un enfoque ingenuo:
1. El usuario presiona "Guardar".
2. La app hace un `fetch` a la API.
3. Si hay red, se guarda. Si no, muestra un toast de error: *"Revisa tu conexión a internet"*.

Esto es inaceptable en operaciones B2B de misión crítica.

## La Solución: SQLite en modo WAL (Write-Ahead Logging)

El enfoque Offline-First invierte la lógica:
1. El usuario presiona "Guardar".
2. La app guarda **inmediatamente** en una base de datos SQLite local en el dispositivo.
3. Un proceso en segundo plano (Background Worker) sincroniza esos datos con la nube cuando detecta conexión.

### Por qué el modo WAL es Crítico

En configuraciones estándar de SQLite, las operaciones de lectura bloquean a las de escritura. En React Native, esto significa que si tu UI está leyendo una gran lista de productos mientras el background worker intenta sincronizar recibos, tu app sufrirá caídas severas de *framerate*.

Habilitando el modo **WAL**, las escrituras se realizan en un archivo de log separado (`-wal`), permitiendo lectores y escritores concurrentes:

```sql
PRAGMA journal_mode=WAL;
```

Con este simple comando (y una arquitectura robusta de control de versiones alrededor), logramos en un caso real de producción **reducir los fallos de sincronización a cero**.
